# Lumen 产品需求文档（2026-04-03）

## 1. 文档目的

本文件用于把当前 `apps/lumen` 从“正在产品化的编辑器壳”收敛为“可持续完善的产品需求基线”。

它解决三件事：

1. 统一当前项目对 Lumen 的产品定位，避免继续把“demo 能跑”和“产品可交付”混为一谈。
2. 把现有代码、路线图、收口文档拆成可执行的需求项、验收项和迭代顺序。
3. 作为后续功能完善、代码拆分、测试补齐和交付评估的唯一产品需求入口。

相关文档：

- `docs/lumen-product-completion-plan.md`
- `docs/editor-gap-roadmap.md`
- `docs/lumen-feature-packaging-plan.md`
- `docs/session-handoff.md`

## 2. 当前阶段判断

### 2.1 当前不是单纯 demo

当前 `apps/lumen` 已经具备以下产品壳能力：

- 文档中心：首页、文档列表、创建文档、账户入口。
- 工作区壳：顶部栏、菜单栏、工具栏、浮动操作、右侧面板。
- 协作链路：后端会话、文档权限、分享链接、协作 ticket、presence。
- 审阅链路：评论、Track Changes、目录、批注。
- 内容能力：分页编辑、表格、图片、视频、模板、导入导出、工具插入。
- 辅助能力：AI 助手、i18n、基础 E2E 回归。

### 2.2 当前也还不是产品完成态

当前仍存在以下中间态特征：

- `apps/lumen` 壳层过重，`App.vue`、`EditorToolbar.vue`、`editorMount.ts` 承担了过多运行时编排责任。
- `playground` 命名和运行时习惯仍残留在产品壳里，说明迁移尚未彻底完成。
- 后端仍是 MVP 形态：元数据默认使用 JSON 文件，协作文档直接文件落盘，不适合作为最终生产方案。
- 编辑器主战场仍在复杂分页、表格、overlay、IME、worker 路径稳定性。
- 仓库基线已恢复到 `pnpm typecheck` 通过，但产品壳拆分、路由主流程和编辑器稳定性仍未收口。

### 2.3 本阶段结论

Lumen 当前阶段定义为：

`分页文档编辑器平台上的产品壳 MVP，目标是向可交付产品收口。`

这意味着后续完善优先级必须服从：

1. 稳定性与权限一致性
2. 产品壳分层收敛
3. 审阅/协作主流程闭环
4. 后端可替换与可部署性
5. 扩展生态和高级能力

## 3. 产品定位

### 3.1 产品目标

Lumen 的目标不是做一个轻量富文本输入框，而是做一个面向分页文档场景的在线文档产品壳，重点覆盖：

- 长文档编辑
- 真分页布局
- 协作编辑
- 审阅与评论
- 权限与分享
- 产品级工具栏与工作区交互

### 3.2 目标用户

1. 内容生产团队
   - 需要稳定编辑长文档、模板文档和媒体富文档。
2. 产品/运营/项目团队
   - 需要多人协作、评论、修订、分享阅读。
3. 内部平台团队
   - 需要在 Lumen 之上继续扩展业务节点、模板和交付流程。

### 3.3 目标场景

1. 内容后台编辑：运营手册、产品文档、规范文档、媒体稿件。
2. 协同审阅：多人评论、修订、只读分享、评论态分享。
3. 模板化文档生产：带页眉页脚、目录、分页设置、导出预览的正式文档。

## 4. 本阶段产品目标

本阶段不追求“一次性做全”，只追求把 Lumen 收口到“可持续交付”的状态。

### 4.1 阶段目标

1. 建立稳定的产品主流程。
2. 建立清晰的产品壳与引擎边界。
3. 建立可重复验证的回归基线。
4. 建立后端可升级的接口与数据边界。

### 4.2 非目标

以下内容不作为本阶段必须完成项：

- 全量对标 Google Docs 的所有细节交互。
- 完整的企业级后台管理系统。
- 复杂组织空间、团队管理、计费、多租户。
- 所有高级扩展一次性产品化。

## 5. 产品范围

### 5.1 P0 必须闭环

1. 文档访问与工作区加载闭环。
2. 权限模式闭环：`full / comment / readonly`。
3. 协作接入闭环：文档 ticket、presence、连接状态。
4. 评论与修订主流程闭环。
5. 分页编辑核心稳定性闭环。
6. 最基本的导入导出和分享闭环。

### 5.2 P1 本阶段应该完成

1. 产品壳拆分与状态收敛。
2. 工具栏动作体验统一。
3. 工作区侧栏与浮动操作一致性。
4. 后端文档中心体验和分享体验收口。
5. 关键流程自动化回归补齐。

### 5.3 P2 可以延后

1. 更完整的插件模板和生态规范。
2. 更丰富的组织/空间能力。
3. 更完整的企业级导出、发布、审批流程。
4. 数据库存储、对象存储、可观测性平台对接。

## 6. 功能需求

## 6.1 文档中心

### 目标

用户可以完成登录、查看文档列表、创建文档、进入工作区。

### 需求

- 提供文档中心首页。
- 支持当前用户会话识别。
- 支持拉取当前用户可访问文档列表。
- 支持创建文档并自动进入工作区。
- 支持多语言文案切换。

### 验收

- 未登录状态下可明确看到登录入口和受限提示。
- 已登录状态下可以看到文档列表。
- 创建文档后能稳定跳转到对应工作区。
- 刷新页面后会话和列表状态行为一致。

## 6.2 工作区加载与路由

### 目标

用户通过文档 ID 或分享链接进入工作区时，系统能稳定解析访问身份、权限、协作状态和文档上下文。

### 需求

- 支持 `/docs/:documentId` 工作区入口。
- 支持 `/share/:token` 分享落地页。
- 支持登录态用户和匿名分享用户的不同访问路径。
- 支持在进入工作区前完成文档访问鉴权。
- 支持通过后端换取协作 ticket。

### 验收

- 无权限访问时给出清晰反馈，不进入错误工作区。
- 分享页进入工作区后，访问身份和权限模式正确继承。
- 页面刷新、重新进入、切换账号后，权限和会话不串。

## 6.3 编辑主流程

### 目标

编辑区需要稳定支撑分页文档的核心编辑体验。

### 需求

- 支持正文、标题、列表、表格、图片、视频、模板、分页设置等基础编辑能力。
- 支持工具栏、菜单栏、气泡菜单、快捷键共同驱动编辑命令。
- 支持目录、页眉页脚、分页、页面设置等分页文档能力。
- 支持导入 Word/Markdown，支持导出 PDF/HTML/Text/Word/Image。

### 验收

- 常用内容节点插入后可见、可编辑、可保存、可再次打开。
- 工具栏行为与当前权限模式一致。
- 分页相关设置影响布局后，编辑区和预览行为一致。
- 导入导出链路可至少覆盖基础用例并可回归。

## 6.4 协作与 Presence

### 目标

多人协作场景下，连接状态、远端用户和权限应可理解、可恢复、可验证。

### 需求

- 支持基于 Yjs/Hocuspocus 的协作连接。
- 支持文档级协作 ticket。
- 支持协作连接状态展示。
- 支持远端用户 presence 展示。
- 支持协作异常时的可见错误反馈。

### 验收

- 两个客户端进入同一文档可看到实时同步结果。
- 远端用户列表和光标状态可更新。
- ticket 失效、权限不足、连接失败时有清晰提示。
- 分享态和成员态的协作权限正确区分。

## 6.5 评论与修订

### 目标

Lumen 必须具备可用于真实审阅的基础评论和修订流程。

### 需求

- 支持选区评论、评论线程、回复、删除、解决状态。
- 支持 Track Changes 开关、定位、接受、拒绝、接受全部、拒绝全部。
- 支持评论与修订面板联动编辑区定位。
- 支持评论态权限下发表评论但不能编辑正文。

### 验收

- 评论锚点可创建、可定位、可随着文档修改保持有效，失效后可清理。
- 修订记录可产生、可聚合、可接受/拒绝。
- `readonly` / `comment` / `full` 三种模式下行为一致。

## 6.6 分享与权限

### 目标

产品壳必须把“谁能看、谁能评、谁能改”变成可见且一致的产品行为。

### 需求

- 支持成员角色：`owner / editor / commenter / viewer`。
- 支持分享链接角色映射到权限模式。
- 支持匿名访问开关。
- 支持工作区内看到当前权限状态。
- 支持前端会话模式和后端权限共同决定最终可操作能力。

### 验收

- owner、editor、commenter、viewer 行为符合预期。
- 分享链接授予的权限不能越权。
- 顶部栏、工具栏、侧栏按钮禁用状态与权限一致。

## 6.7 AI 助手

### 目标

AI 是产品辅助能力，不应破坏编辑主流程和权限边界。

### 需求

- 提供 AI 面板和基础供应商配置。
- 仅在允许编辑的上下文开放内容修改能力。
- AI 结果应用到文档时应能感知当前编辑器状态。
- AI 请求通过后端代理，避免直接暴露密钥。

### 验收

- 未配置 AI 时给出明确提示。
- 配置正确时可完成一次内容处理并写回文档。
- 权限不足时不提供越权写入能力。

## 6.8 工具栏与菜单体验

### 目标

工具栏需要从“动作接线完成”收口到“产品交互完成”。

### 需求

- 工具栏、菜单栏、导出入口遵循统一的产品策略。
- 禁止依赖原生 `prompt/alert` 作为主流程交互。
- 禁止仅用纯文本占位替代真实节点能力。
- 高级插入能力必须给出明确的能力边界和降级策略。

### 验收

- 接线完成不再作为唯一完成标准。
- 每个产品能力至少满足：可触发、可见结果、权限正确、可回归。
- 产品策略变更后，文档与目录配置同步更新。

## 7. 非功能需求

## 7.1 稳定性

- 全仓 `pnpm typecheck` 必须通过。
- `apps/lumen` 关键 E2E 用例必须可运行。
- 已知主线回归项不得长期停留在手工验证状态。

当前已知问题：

- `typecheck` 已恢复通过，后续改动不得重新引入跨 workspace 的类型断链。

## 7.2 性能

- 长文档分页和输入不能出现明显卡顿和光标乱跳。
- worker 路径和主线程路径行为必须一致。
- 页复用、虚拟化、增量布局不能以破坏正确性为代价。

## 7.3 安全

- 粘贴、HTML、Markdown、JSON 导入都必须经过统一安全策略。
- 链接、图片、视频等外部资源必须遵循统一协议白名单。
- AI、协作和分享接口不能绕过权限检查。

## 7.4 可维护性

- 产品壳状态管理要从巨型文件中拆开。
- 可复用逻辑优先下沉到 `core / engine / extensions`，但不把产品节奏硬塞到底层。
- 新功能必须说明其所属层级：`app / extension / core / engine / backend`。

## 7.5 状态管理策略

- `apps/lumen` 需要正式引入状态管理库，当前选择 `Pinia` 作为产品壳状态容器。
- `Pinia` 只管理产品壳状态：工作区访问、会话用户、分享/账户弹层、页面级 loading/error、侧栏与壳层交互状态。
- 编辑器内核状态不进入 `Pinia`：文档、选区、评论锚点、修订、协作同步仍由 editor runtime、插件状态和 Yjs 托管。
- 远程数据状态优先先通过后端访问 composable 编排；当文档中心、成员、分享链接页面继续扩张时，再评估补充 query/cache 层。
- 新增跨页面或跨壳层共享状态时，默认优先进入 `Pinia store`，而不是继续堆进 `App.vue` 或单组件 composable。

## 8. 工程边界要求

### 8.1 允许保留在 `apps/lumen` 的内容

- 工具栏动作编排与交互节奏
- 会话模式与权限体验
- 分享弹层、账户弹层、产品工作区 UI
- 产品侧文案、流程编排和入口策略

### 8.2 应该下沉或收敛的内容

- 可复用的编辑器扩展装配能力
- 可复用的权限插件、协作适配、导入导出策略
- 独立可测试的命令逻辑和节点能力
- 与 Vue 壳层无关的运行时逻辑

### 8.3 后端边界

后端本阶段需保持“接口稳定、实现可替换”的原则：

- 前端不依赖 JSON 存储实现细节。
- 协作文档持久化方案可替换。
- 分享、权限、ticket、AI 代理接口保持稳定。

## 9. 里程碑与迭代顺序

## M0 基线收口

### 目标

建立继续开发的健康基线。

### 任务

1. 保持全仓 `typecheck` 持续为绿色基线。
2. 固定本需求文档为产品入口。
3. 同步 `session-handoff` 阅读顺序。
4. 把当前 P0 风险列成明确清单。

### 验收

- `pnpm typecheck` 通过。
- 团队继续开发时先读本文件而不是仅凭零散 handoff。

## M1 产品壳拆分

### 目标

降低 `apps/lumen` 工作区壳的耦合度。

### 任务

1. 拆分 `App.vue` 中的会话、权限、协作、评论、修订编排逻辑。
2. 收敛 `editorMount.ts` 的运行时职责。
3. 清理 `playground` 命名残留。
4. 统一页面壳与工作区状态流。

### 验收

- `App.vue` 不再承担全部产品状态编排。
- 主要状态有清晰模块边界。

## M2 主流程稳定化

### 目标

把用户真正会走的主链路做稳。

### 任务

1. 收口权限模式行为。
2. 收口分享进入工作区行为。
3. 补评论与修订回归。
4. 收口表格复杂分页、overlay、IME、worker 主问题。

### 验收

- 主路径可稳定演示和回归。
- 当前 `session-handoff` 中的主线问题显著减少。

## M3 交付能力完善

### 目标

让 Lumen 具备更明确的产品交付形态。

### 任务

1. 收口导入导出体验。
2. 收口文档中心、分享、工作区之间的产品一致性。
3. 抽象后端存储层，为数据库替换做准备。
4. 明确部署、配置和联调文档。

### 验收

- 能以“单文档产品壳”形态稳定联调和展示。
- 后端不再强依赖当前最简持久化实现。

## 10. DoD

一个需求项只有同时满足以下条件才算完成：

1. 有明确产品入口。
2. 有真实结果，而不是占位或伪交互。
3. 权限模式行为正确。
4. 有自动化或可重复人工回归路径。
5. 文档同步更新。

## 11. 当前建议执行顺序

1. 先守住基线健康状态，尤其是 `typecheck`、路由主流程和明显断链问题。
2. 再拆 `apps/lumen` 壳层，把会话、权限、协作、评论、修订编排从巨型组件中抽出。
3. 然后收口用户主流程：文档进入、分享进入、评论、修订、协作状态。
4. 再解决复杂编辑稳定性问题：table、overlay、IME、worker。
5. 最后推进后端可替换性和更完整的交付链路。

## 12. 后续维护约定

- 本文件是 Lumen 产品需求基线。
- 新增产品能力时，先补本文件中的范围和 DoD，再开始实现。
- 路线图文档可以继续保留，但产品优先级以本文件为准。
